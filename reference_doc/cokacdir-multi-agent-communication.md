# cokacdir: 텔레그램 다중 봇·에이전트(LLM) 간 소통 구조

이 문서는 cokacdir 저장소(`src/services/telegram.rs`, `src/main.rs` 등)의 구현을 기준으로, **하나의 텔레그램 그룹에 여러 봇을 두었을 때 에이전트 간에 정보가 어떻게 오가는지**를 정리한 것이다.

---

## 1. 요약

- **LLM 인스턴스끼리 직접 연결된 통신 채널은 없다.** 각 봇은 독립된 세션·모델·작업 디렉터리로 동작한다.
- 소통은 크게 세 가지 축으로 나뉜다.
  1. **텔레그램 그룹**: 사용자가 보낸 메시지가 (설정·라우팅 규칙에 따라) 여러 봇에 동시에 전달되거나, 특정 봇만 타겟팅된다.
  2. **공유 그룹 로그(JSONL)**: 모든 봇의 user/assistant 기록이 한 파일에 쌓이고, 다음 요청의 **시스템 프롬프트**에 최근 줄이 주입되어 **간접적으로** 다른 봇의 맥락을 본다.
  3. **`cokacdir --message`**: 로컬 디렉터리에 JSON 메시지 파일을 써서 **봇→봇 1:1**로 넘기고, 수신 봇이 이를 `[BOT MESSAGE from …]` 형태의 사용자 프롬프트로 LLM에 넣는다.

---

## 2. 다중 봇 프로세스 구조

`--ccserver`에 **봇 토큰을 여러 개** 넘기면, 각 토큰마다 `run_bot`이 **별도 비동기 태스크**로 실행된다 (`src/main.rs`의 `handle_ccserver`).

- 한 그룹에 봇이 3개여도, 서버 쪽에서는 **토큰(인스턴스)마다 폴링·상태가 분리**된다.
- 각 인스턴스는 자기 **토큰 해시 키**로 `~/.cokacdir/bot_settings.json` 등에 설정을 나눠 저장한다.

---

## 3. 사용자 ↔ 여러 봇 (그룹 채팅 라우팅)

구현·문서(`website`의 그룹 채팅 가이드) 기준 요지는 다음과 같다.

| 패턴 | 대상 |
|------|------|
| `@봇이름 메시지` | 해당 봇만 처리 (다른 봇은 무시) |
| `;메시지` | **모든 봇**이 (메시지가 각 봇에 도달하는 한) 동일 사용자 문장을 각자 처리 |
| `/명령@봇이름` | 해당 봇만 |
| `/명령` (bare) | 텔레그램이 **임의의 한 봇**에만 전달할 수 있음 |

**텔레그램 Privacy Mode**가 켜져 있으면 `/`와 `@멘션` 위주로만 봇에 전달되고, `;` 접두 메시지 등은 봇이 **아예 수신하지 못할 수 있다.** 멀티봇·`;` 사용 시 문서에서는 **Disable** 권장.

그룹에서는 일반 평문은 무시하고, `;` / `@` / `/` 등 **접두 규칙**을 만족할 때만 처리하는 필터가 있다 (`handle_message`의 `prefix_filter`).

---

## 4. 공유 그룹 로그 (간접 엿듣기)

### 4.1 위치·역할

- 경로: `~/.cokacdir/group_chat/<chat_id>.jsonl`
- **같은 그룹 채팅 ID**에 속한 모든 봇이 여기에 **user / assistant** 항목을 append 한다.
- 코드 주석상 목적: *“각 봇은 자기 대화만 보이므로, cross-bot context 문제를 풀기 위해 공유 로그를 둔다.”*

### 4.2 LLM에 어떻게 들어가나

- `/context`로 설정한 **최근 N줄**이 0보다 크면, `build_system_prompt`가 **GROUP CHAT LOG** 섹션을 시스템 프롬프트에 넣는다.
- 따라서 봇 B가 남긴 assistant 기록이, 봇 A의 **다음 요청**에서 시스템 쪽 맥락으로 반영될 수 있다 (같은 호스트·같은 `~/.cokacdir` 전제).

### 4.3 중복 답변 완화

- 동일한 **사용자 문장**이 이미 **다른 봇**의 user 로그로 기록된 경우, “다른 봇이 이미 같은 요청에 답했다”는 **dedup 경고**를 넣어 **같은 답 반복·같은 작업 재수행**을 피하도록 유도한다.

---

## 5. 봇→봇 직접 메시지: `--message`

### 5.1 개념

- 텔레그램 채팅에 `@상대봇`만 쓴다고 **상대 봇의 LLM 세션에 자동 전달되지 않는다**고 시스템 프롬프트에 명시되어 있다.
- 봇끼리 직접 주고받는 경로는 **`cokacdir` CLI**:
  - `cokacdir --message "<내용>" --to <봇유저명> --chat <채팅ID> --key <발신 봇 토큰 SHA 해시>`
- `main.rs`의 `handle_bot_message`가 `~/.cokacdir/messages/` 아래 **JSON 파일**을 생성한다.

### 5.2 수신 처리

- 각 봇은 `scan_messages`로 `to` 필드가 **자기 username**인 파일을 모은다.
- 처리 시 사용자 메시지에 가깝게  
  `[BOT MESSAGE from …]`  
  형태로 LLM에 넘긴다 (`process_bot_message`).

### 5.3 설계상 주의 (프롬프트에 포함된 가이드)

- 상대의 **도구 사용 내역**은 `--message` 본문에 없을 수 있으므로, 필요 시 **`--read_chat_log`** 로 상대 봇 로그를 보라고 안내한다.
- `--message`를 남발하면 봇↔봇 대화가 **끝나지 않을 수 있어**, “대화를 끝내려면 `--message`를 호출하지 말고 채팅에만 최종 답을 올려라” 등의 **종료 규칙**이 프롬프트에 있다.

---

## 6. 동시성: 그룹 채팅 락

- `chat_id < 0`(그룹)일 때 `acquire_group_chat_lock`으로 **파일 락**을 잡는다.
- 같은 그룹에서 **AI 요청이 동시에 여러 개** 돌아가며 공유 로그·세션과 충돌하는 것을 줄인다.

---

## 7. 한 줄로 정리

| 경로 | 의미 |
|------|------|
| 텔레그램 UI | 사용자·멘션·`;`·명령으로 **여러 LLM에 같은/다른 입력** 분배 |
| `group_chat/*.jsonl` | **모든 봇 대화 스냅샷** → 다음 요청의 **시스템 프롬프트**에 반영 |
| `messages/*.json` + LLM 호출 | **봇→봇 전용 사설 채널** (`[BOT MESSAGE …]`) |

**직접적인 LLM-to-LLM 소켓/공유 컨텍스트 윈도우는 없고**, 디스크·텔레그램 API·CLI를 통한 **간접 오케스트레이션**이다.

---

## 8. 참고 코드 위치 (cokacdir 저장소)

| 내용 | 파일 |
|------|------|
| 다중 토큰으로 여러 `run_bot` | `src/main.rs` (`handle_ccserver`) |
| 그룹 공유 로그, `build_system_prompt`, dedup | `src/services/telegram.rs` |
| `--message` 처리 | `src/main.rs` (`handle_bot_message`) |
| 메시지 스캔·`[BOT MESSAGE …]` | `src/services/telegram.rs` (`scan_messages`, `process_bot_message`) |
| 그룹 락 | `src/services/telegram.rs` (`acquire_group_chat_lock`) |
