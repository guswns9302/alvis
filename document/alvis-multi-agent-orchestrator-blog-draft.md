# 여러 AI 에이전트를 한 팀처럼 운영하려면 무엇이 필요할까

`alvis`를 만들면서 먼저 한 일은 코드를 쓰는 것이 아니라, 이미 비슷한 문제를 풀고 있는 시스템들이 에이전트를 어떻게 묶고 운영하는지 조사하는 일이었다. 특히 `cokacdir`와 `Ensemble`을 보면 공통된 결론이 하나 나온다. 여러 AI 에이전트가 동시에 일한다고 해서, 그들이 어떤 신비한 공유 지능 위에서 직접 대화하는 것은 아니라는 점이다.

현실의 멀티 에이전트 시스템은 훨씬 더 투박하다. 각 에이전트는 독립된 세션으로 실행되고, 메시지는 파일이나 HTTP, 혹은 입력 주입 같은 단순한 메커니즘으로 전달된다. 중요한 것은 "에이전트들이 얼마나 똑똑한가"보다 "그들을 어떤 제어 평면 위에서 안정적으로 운영하는가"에 가깝다.

이 글은 그 조사 내용을 바탕으로, 왜 `alvis`를 `LangGraph + tmux + Codex CLI` 구조로 설계했는지 정리한 초안이다.

## 출발점: 에이전트는 생각보다 독립적이다

처음에는 멀티 에이전트 시스템을 떠올리면 에이전트끼리 실시간으로 대화하고, 서로의 상태를 자연스럽게 공유하는 그림을 상상하기 쉽다. 하지만 실제 구현은 그렇지 않았다.

`cokacdir`에서는 여러 봇이 하나의 그룹에 들어가더라도, 각 봇의 LLM 세션은 서로 직접 연결되어 있지 않다. 사용자 입력은 텔레그램 규칙에 따라 각 봇에게 분배되고, 최근 대화는 JSONL 로그를 시스템 프롬프트에 다시 넣는 방식으로 간접 공유된다. 필요하면 `--message` 같은 별도 경로로 봇 간 메시지를 파일에 기록해 전달한다.

`Ensemble`도 비슷하다. 에이전트는 각자 `tmux` 세션에서 실행되지만, 대화의 진실 원본은 `tmux`가 아니다. 메시지는 JSONL 파일에 append되고, 브리지 프로세스가 이를 HTTP API로 넘기며, 서버가 각 에이전트의 `tmux` 세션에 입력을 다시 주입한다. 즉, `tmux`는 협업의 본체가 아니라 실행 화면이자 입력 채널에 가깝다.

이 두 사례를 보고 나니 한 가지 원칙이 명확해졌다.

## 핵심 원칙: 화면과 상태 저장소를 분리해야 한다

멀티 에이전트 시스템에서 가장 위험한 설계는 터미널 화면을 곧 시스템 상태로 취급하는 것이다. 화면은 보기에는 편하지만, 복구와 재시작, 자동화, 재현성 관점에서는 매우 약하다.

터미널 pane만 믿으면 이런 문제가 생긴다.

- 어떤 작업이 실제로 시작됐는지 기계적으로 판단하기 어렵다.
- 에이전트가 멈췄는지, 단순히 출력이 없는지 구분하기 어렵다.
- 프로세스 재시작 후 상태를 복원하기 어렵다.
- 사람이 보고 이해하는 흐름과, 시스템이 판단해야 하는 흐름이 뒤엉킨다.

그래서 `alvis`에서는 `tmux`를 "실행과 관찰을 위한 UI"로 두고, 상태의 진실 원본은 별도 저장소에 두는 방향을 택했다. 실제 팀 상태, 작업 상태, 이벤트 로그, 리뷰 요청, 세션 메타데이터는 DB와 런타임 파일에서 복원 가능해야 한다. `tmux`는 그 위에 올라간 실행 인터페이스일 뿐이다.

이 구분은 사소해 보여도 운영 난이도를 크게 바꾼다. 화면을 잘 보여주는 것과, 시스템을 안정적으로 운영하는 것은 전혀 다른 문제이기 때문이다.

## 왜 `tmux`를 버리지 않았는가

그렇다면 아예 `tmux`를 쓰지 않으면 되지 않느냐는 질문도 가능하다. 하지만 로컬에서 여러 `Codex CLI` 세션을 동시에 돌리고, 사람이 그 진행 상황을 빠르게 관찰해야 하는 환경에서는 `tmux`가 여전히 강력하다.

`tmux`는 다음 역할에 집중할 때 가장 가치가 크다.

- 에이전트별 독립 터미널 세션 실행
- leader와 worker의 시각적 분리
- 특정 세션에 안전하게 텍스트 입력 주입
- 장애 상황에서 pane 상태를 빠르게 점검

반대로 `tmux`가 해서는 안 되는 일도 분명하다.

- 대화 이력의 영속 저장소 역할
- 작업 상태의 source of truth 역할
- 메시지 라우팅 버스 역할
- 워크플로 의사결정 엔진 역할

조사한 시스템들을 보면, 잘 돌아가는 구조는 대부분 이 선을 지키고 있었다. `tmux`는 실행 UI이고, 오케스트레이션은 다른 계층이 맡는다.

## `LangGraph`를 제어 평면으로 둔 이유

`alvis`에서는 그 "다른 계층"을 `LangGraph`가 맡는다. 이유는 단순하다. 멀티 에이전트 운영의 핵심 문제는 대화 생성이 아니라 상태 전이 관리이기 때문이다.

실제 운영에서는 계속 이런 판단이 필요하다.

- 지금 요청을 몇 개의 task로 나눌지
- 어떤 worker에게 어떤 범위를 맡길지
- 결과가 불완전할 때 redo를 만들지, 다른 역할로 handoff할지
- 사람 확인이 필요한 시점이 언제인지
- 전체 run을 언제 종료할지

이런 흐름은 일회성 프롬프트보다 상태 그래프에 가깝다. 그래서 `LangGraph`를 제어 평면으로 두면, task 분해, dispatch, 재시도, 타임아웃, handoff, interrupt 같은 흐름을 명시적으로 다룰 수 있다.

정리하면 `alvis`의 역할 분담은 이렇게 된다.

- `LangGraph`: control plane
- `tmux`: execution UI
- `Codex CLI`: 실제 작업을 수행하는 actor
- `SQLite`와 런타임 파일: 상태와 이벤트의 source of truth

이 구조의 장점은 책임 경계가 분명하다는 점이다. 어디서 판단하고, 어디서 실행하고, 어디서 복구할지를 분리해야 시스템이 커져도 무너지지 않는다.

## 에이전트는 "대화 상대"보다 "장기 실행 actor"에 가깝다

이번 조사에서 또 하나 중요했던 관점은, 에이전트를 채팅창 안의 말하는 존재로 보기보다 "계약을 받아 실행하고 결과를 돌려주는 actor"로 보는 편이 더 실용적이라는 점이다.

`alvis`에서 각 `Codex CLI` 세션은 다음 성격을 가진다.

- 자기 컨텍스트와 작업 디렉터리를 가진다.
- 명시적인 task contract를 입력으로 받는다.
- 제한된 쓰기 범위와 역할을 가진다.
- 구조화된 결과를 다시 보고한다.

이 관점으로 보면 멀티 에이전트 협업은 감각적인 "대화 orchestration"이 아니라, 꽤 전통적인 분산 작업 실행 문제에 가까워진다. 누가 어떤 범위를 맡고, 어떤 포맷으로 완료를 선언하며, 실패 시 누가 다시 이어받는지가 더 중요해진다.

그래서 `alvis`는 shared root를 쓰더라도 `owned_paths` 같은 범위 계약을 두고, worker 결과도 자유 형식 텍스트보다 구조화된 결과 블록으로 수집하도록 설계했다. 에이전트가 많아질수록 이런 계약이 없으면 협업은 금방 흐트러진다.

## 사람이 개입해야 하는 순간도 설계 대상이다

멀티 에이전트 시스템을 자동화한다고 해서 사람을 완전히 빼는 것이 목표는 아니다. 오히려 어떤 순간에 사람 승인이 필요한지를 1급 기능으로 다루는 편이 더 현실적이다.

예를 들어 다음 작업은 자동화보다 승인 지점이 중요하다.

- `git commit`
- `git push`
- 대량 삭제나 파일 이동
- 계획 변경 폭이 큰 재작업

조사 과정에서도 느꼈지만, 자동화 시스템이 신뢰를 얻으려면 "계속 스스로 움직이는 능력"만큼 "멈춰야 할 때 정확히 멈추는 능력"이 중요하다. 그래서 `alvis`는 사람 승인 지점을 흐름 바깥 예외 처리로 두지 않고, 오케스트레이션 설계 안에 넣으려 한다.

## 결국 중요한 것은 멀티 에이전트가 아니라 운영 가능성이다

겉으로 보면 멀티 에이전트 시스템의 핵심은 여러 모델을 동시에 일하게 만드는 데 있는 것처럼 보인다. 하지만 실제로는 그보다 운영 가능성이 더 중요하다.

- 실패했을 때 다시 붙일 수 있는가
- 누가 어떤 작업을 했는지 추적 가능한가
- 잘못된 결과를 자동으로 걸러낼 수 있는가
- 사람이 현재 상태를 이해할 수 있는가
- 상태 저장소만으로 시스템을 복원할 수 있는가

`cokacdir`와 `Ensemble` 조사에서 얻은 가장 큰 교훈도 여기에 있다. 좋은 멀티 에이전트 시스템은 화려한 "에이전트 간 대화"보다, 투박하지만 복원 가능한 메시지 경로와 명시적인 상태 관리 위에 세워진다.

`alvis`는 아직 초기 단계지만, 적어도 방향은 분명하다. `tmux`는 화면으로 남기고, `LangGraph`는 제어 평면으로 두고, 에이전트는 장기 실행 actor로 다룬다. 그 위에서 review, redo, handoff, recovery를 점차 더 정교하게 만드는 것이 다음 과제다.

## 마무리

이번 조사는 "멀티 에이전트 시스템은 어떻게 에이전트끼리 대화하게 만들까?"라는 질문을, "멀티 에이전트 시스템을 어떻게 운영 가능한 구조로 만들까?"라는 질문으로 바꾸게 만들었다.

이 관점 전환이 `alvis` 설계의 출발점이었다. 앞으로 구현이 더 쌓이면, 실제 실행 화면과 상태 저장소가 어떻게 연결되는지, 자동 handoff와 redo가 어떤 기준으로 움직이는지, 그리고 로컬 환경에서 이런 팀형 AI 워크플로가 어디까지 실용적인지까지 이어서 정리해볼 생각이다.

## 초안 메모

- 현재 버전은 설계 배경과 아키텍처 원칙 중심의 블로그 초안이다.
- 이후 보강할 만한 항목은 실제 실행 화면 캡처, `leader 1 + worker 2` 흐름 예시, 실패 복구 사례다.
- 외부 공개용으로 다듬을 때는 `cokacdir`, `Ensemble`, `alvis` 각 구조를 비교하는 도표를 추가하면 이해가 더 빨라진다.
